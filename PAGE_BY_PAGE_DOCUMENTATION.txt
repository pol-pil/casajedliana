==================================================
1. DASHBOARD PAGE
==================================================

Route:
- `/dashboard`

Frontend file:
- `resources/js/pages/Dashboard/Index.tsx`

Backend file:
- `app/Http/Controllers/DashboardController.php`

Rendered Inertia page:
- `Dashboard/Index`

Purpose:
- gives frontdesk staff a live operational overview for the selected date range

Server-side data flow:
- controller reads `start` and `end` query parameters
- defaults to today and tomorrow when parameters are missing
- loads rooms with conditional booking overlap counts
- loads active bookings, check-ins, and check-outs
- maps bookings through `refreshPaymentStatus()`
- returns props:
  - `startDate`
  - `endDate`
  - `rooms`
  - `bookings`
  - `checkIns`
  - `checkOuts`

Important frontend state:
- `roomScope`
- `activeFilter`
- `search`

Important frontend hooks/utilities:
- `usePage`
- `useDateRange`
- `useMemo`
- `useState`
- Inertia `router`
- `date-fns/format`

Important page logic:
- `bookingOverlapsRange()` checks whether a booking overlaps the selected date context
- `occupiedRoomIds` is memoized from overlapping bookings
- `availableCount` is derived from room list minus occupied room IDs
- room list is split by scope:
  - hotel rooms
  - event rooms
- room list is filtered by status tab before rendering

Important UI sections:
- summary cards for check-ins, check-outs, unavailable, available
- room scope toggle
- room status tabs
- room status grid
- quick action card
- arrival/departure table

Important components used:
- `AppLayout`
- `Head`
- `Card`
- `CardContent`
- `CardHeader`
- `CardTitle`
- `Badge`
- `Input`
- `Button`
- `Tabs`
- `TabsList`
- `TabsTrigger`

Important data shown:
- room number
- room type
- room status
- guest full name
- check-in date/time
- check-out date/time
- booking status
- payment badge

Important navigation/actions:
- `router.visit('/bookings?create=true')`
- `router.visit('/reports/history')`

Technical notes:
- room status is normalized on the server to `Available`, `Reserved`, `Occupied`, or `Maintenance`
- selected date range comes from shared context rather than a page-local picker
- the page uses derived room occupancy rather than directly trusting persisted display status for the chosen range

2. BOOKINGS PAGE
==================================================

- `/bookings`

- computes `stats`
- loads supporting datasets:

  - `rates`
  - `charges`
  - `bookingTypes`
  - `clients`
  - `roomBlockedDates`
- returns props:
  - `bookings`
  - `calendarBookings`
  - `stats`
  - `rooms`
  - `rates`
  - `charges`
  - `bookingTypes`
  - `clients`
  - `roomBlockedDates`
  - `filters`

Important frontend state:
- `isDialogOpen`
- `dateRange`
- `selectedRoomId`
- `guestCount`
- `selectedRateId`
- `selectedBookingType`
- `isBookingInfoDialogOpen`
- `selectedBooking`
- `checkInTime`
- `checkOutTime`
- `isEditMode`
- `editingBookingId`
- `activeTab`
- `roomScope`
- `search`

Important frontend hooks/utilities:
- Inertia `useForm`
- Inertia `usePage`
- Inertia `router`
- `useEffect`
- `useMemo`
- `useRef`
- `toast` from `sonner`

Important page logic:
- auto-opens booking form when current URL contains `create=true`
- debounced guest search triggers `router.get('/bookings', ...)`
- same dialog handles create and edit flows
- selected booking is refreshed when props update
- booking table is filtered locally by active tab after server filtering
- calendar resources are generated from room list
- calendar events are generated from booking list

Important UI sections:
- room/resource timeline
- booking statistics cards
- search + status tabs row
- floating new booking button
- bookings table
- pagination controls
- booking info modal
- booking form modal

Important components used:
- `AppLayout`
- `ResourceTimeline`
- `BookingFormDialog`

- `Tabs`
- `Input`
- `Button`
- `Badge`

Important data shown:
- guest name
- guest email
- guest contact number
- room number
- room type
- check-in and check-out timestamps
- booking status
- total amount
- computed balance
- payment status in calendar event chips

Important server actions/endpoints tied to the page:
- `POST /bookings`
- `PUT /bookings/{booking}`
- `PATCH /bookings/{booking}/status`
- `GET /bookings/{booking}/print`

Important backend logic:
- `RoomPricing` service is injected into controller constructor
- custom discounts can create a custom `Rate` record dynamically
- client is created with `firstOrCreate()` using contact number
- booking totals use computed pricing breakdown
- room status is synced through `syncRoomStatus()`
- activity logs are inserted for create/update/status changes
- payment status is recalculated after downpayment and updates

Technical notes:
- page mixes paginated table data and non-paginated calendar data
- timeline rendering depends on helper theme functions in `fullcalendar-theme.ts`
- booking balance includes booking charges minus payments
- modal editing preloads form data from selected booking object

- room description is composed from bed type plus amenities
- filtered rooms support:
  - keyword search
  - status filtering
  - category filtering
- filtered rooms are regrouped by category before rendering
- date changes trigger `router.get('/rooms', { start, end })`

Important UI sections:

- room search input
- filter controls
- grouped room category sections
- room cards
- shared room dialogs container

- `AppLayout`
- `RoomHeader`
- `RoomFilters`
- `RoomCard`
- `RoomDialogs`

Important data shown:
- room number
- room category
- room capacity
- weekday rate
- weekend rate
- status
- description/beds

Important server actions/endpoints tied to the page:
- `POST /rooms`
- `PATCH /rooms/{room}`
- `DELETE /rooms/{room}`
- `PATCH /rooms/{room}/status`

Important backend logic:
- create/update uses dedicated form requests:
  - `StoreRoomRequest`
  - `UpdateRoomRequest`
- maintenance status is blocked when an active booking exists
- delete is blocked when bookings already exist for the room
- room `price` is synced from weekday rate on save

Technical notes:
- breadcrumbs in this page point to admin context even though route is `/rooms`
- room category labels are frontend-derived from `room_type`
- the page keeps server date state and local date state aligned
